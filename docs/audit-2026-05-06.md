# Audit Report 2026-05-06

Scope:

- Solidity contracts in `contracts/`
- Deployment and config code in `scripts/` and `src/`
- Tests, docs, pnpm package management, and dependency lockfile

Verification run locally:

- `pnpm exec hardhat compile --force`
- `pnpm run typecheck`
- `pnpm test`
- `pnpm run test:e2e:anvil`
- `pnpm run deploy:token -- --help`
- `pnpm audit --registry=https://registry.npmjs.org --audit-level low`

## Summary

No critical contract issue was found. The main risks are supply-chain/tooling audit findings, operational safety around `isTest`, future UUPS upgrade safety, and deployment script edge cases.

## Findings

### High

1. Dev dependency audit findings in Hardhat toolbox transitive dependencies.

   `pnpm audit` against npm registry reports high advisories for `glob@10.4.5`, `serialize-javascript@6.0.2`, and `lodash-es@4.17.21`. These are in dev/tooling paths through `@nomicfoundation/hardhat-toolbox-mocha-ethers`, Mocha, and Ignition. `pnpm audit --prod` reports no production dependency vulnerabilities.

   Recommendation: track upstream Hardhat toolbox releases, consider pinning/overriding patched transitive versions only after verifying Hardhat compatibility, and run audit against the npm registry in CI because the current npmmirror registry does not support the audit endpoint.

### Medium

1. UUPS upgrade compatibility is not enforced by tests or storage layout checks.

   Current tests upgrade to the same implementation class and check the implementation slot/name, but do not prove future implementations preserve ERC20, Ownable, UUPS, and custom ERC-7201 storage layout.

   Recommendation: emit `storageLayout`, add V2 mock upgrade tests, and require storage layout review before real upgrades.

2. `isTest: true` can be deployed on any chain.

   This is intentional for test tokens, but there is no chain allowlist or confirmation guard. A config mistake can deploy a public unlimited mint token on a production network.

   Recommendation: block `isTest: true` on known mainnet chain IDs unless an explicit escape hatch is present.

3. Upgradeable deployment can leave orphan implementations.

   The script deploys and confirms the implementation before deploying the proxy. If proxy deployment or initialization fails, no structured deployment record is written.

   Recommendation: write partial/failure records for implementation deployments or preflight initializer/proxy deployment more aggressively.

4. `verify.enabled: true` does not verify or fail.

   The script only warns and records `verified: false`, which can mislead CI or operators.

   Recommendation: either fail fast when verification is requested but unsupported, or implement explorer verification before allowing the flag.

5. Deployment script has no automated integration test.

   Unit tests cover contracts and config parsing, but not CLI parsing, `.env` loading, chain ID checks, artifact loading, direct deployment, proxy deployment, and deployment record writing.

   Follow-up: `e2e/anvil.e2e.ts` now covers the main Anvil deployment path, including CLI config parsing, `.env` private key loading, chain ID mismatch, direct deployment, UUPS proxy deployment, deployment records, public mint, and production mint revert. Remaining recommendation: add CI and expand failure recovery, gas, and verify cases.

6. Missing CI quality gate.

   There is no `ci` script or GitHub Actions workflow for frozen install, compile, typecheck, and tests.

   Recommendation: add `pnpm run ci` and a workflow.

### Low

1. Config accepts zero addresses and fails later in contract deployment.
2. `initialSupply` is not explicitly checked against `uint256` range in config parsing.
3. Gas config does not check `maxPriorityFeePerGasGwei <= maxFeePerGasGwei`.
4. `confirmations: 0` is accepted but deployment waits at least 1 confirmation while records still say 0.
5. JSON Schema and zod validation are not fully equivalent.
6. Deployment records for non-local chains are not ignored by default and can include deployer/owner/recipient/metadata.
7. README quick-start order can lead users to run deploy before copying `.env` and `config/tokenconfig.json`.
8. `metadata` is written verbatim to deployment records; user-provided secrets in notes would be leaked if records are committed.
9. `pnpm ignored-builds` reports `esbuild`; current local `tsx` works, but clean machines may need build approval.

## Passed Checks

- ERC20 standard behavior and configured decimals match the spec.
- Production `mint` reverts; test-token public mint works as requested.
- UUPS initializer is single-use and upgrade authorization is owner-only.
- ERC-7201 custom storage slot constant was independently recalculated and matches the implementation.
- Private key is read from `.env`/environment and is not printed or written to deployment records.
- Full RPC URL is not written to deployment records; only `rpcHost` is stored.
- `packageManager` is pinned to `pnpm@10.12.1`, direct dependencies are exact pinned, and OpenZeppelin/Hardhat peer dependencies are currently compatible.
